## 2024-05-24 - Unvalidated Electron openExternal Calls & Synchronous Promise Return Issue
**Vulnerability:** The Electron main process accepted arbitrary URLs via `IPC.OPEN_EXTERNAL` and external window navigation, passing them directly to `shell.openExternal`. This allows for opening dangerous protocols like `javascript:` or `file:`, posing severe remote code execution and local file access risks. Additionally, the return value of `shell.openExternal` was checked synchronously, but it returns a `Promise<void>`, causing an always-truthy check that hid potential errors and prevented intended retry logic.
**Learning:** `shell.openExternal` is inherently dangerous if unchecked and should never be used without an explicitly allowlisted set of protocols (e.g., `http:`, `https:`, `mailto:`). Also, assuming synchronous behavior for asynchronous Electron APIs can mask logic flaws.
**Prevention:** Centralize external link processing to validate protocols explicitly through `isValidExternalUrl`. Enforce this across all points where URLs are opened. Ensure proper asynchronous handling by resolving or awaiting promises where required.
